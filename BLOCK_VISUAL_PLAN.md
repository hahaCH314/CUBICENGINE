# ブロックの見た目 計画書（ヒマワリ用）

> シオン → ヒマワリ。前回シオンが「古代の石板」を**文字通りに作りすぎた**。それは違った。下が正しい方向。
> **細かい数値（px）は指定しない。ヒマワリの感覚で良い。** その代わり「絶対ルール」と「完成チェック」は守って。

---

## 1. いちばん大事（判断のものさし）

非プログラマの**子どもが、自分でマイクラのアドオンを作る**アプリ。ブロックを組んで作る。
迷ったら、いつもこの一言で決める:

> **「アドオンを作れない子どもが、触って楽しいか？」**

狙う気持ち ＝ **ワンピースみたいな"冒険のテンション"（ワクワク・高揚感）**。
※「ワンピース」「ポーネグリフ」は**気分・テンションの例え**。**見た目を石板や遺跡にする意味ではない**（←前回ここを誤解した）。

## 2. 見た目の方向（ここ前回ミスったので最重要）

- ❌ 石板・古代遺跡・ポーネグリフ**みたいな見た目にはしない**。
- ✅ **ブロックはブロックのまま** ＝ マイクラっぽい**可愛いブロック**。
- ✅ ブロックに**名前を可愛く書く、または埋め込む（彫り込み）**。

## 3. 絶対ルール（DO / DON'T）★外したらやり直し

**やる (DO)**
- ✅ 色は今のまま（緑=条件 / 青=アクション / 赤=イベント …カテゴリ色）。
- ✅ **マイクラっぽい可愛いブロック**（角ばった直方体でOK＝カチカチした感じ）。
- ✅ 名前は**可愛く書く、または埋め込む（彫り込み＝沈んで見える）**。読めること。

**やらない (DON'T)**
- 🚫 石板・遺跡っぽくしない。
- 🚫 ブロックを**丸くしない**（丸いとカチカチ感が消える）。
- 🚫🚫 **平たい四角いラベルにしない（最重要・今ここで失敗した）**。ブロックは**3Dの立体＝上の面と横の面が見える"厚みのあるキューブ"**。小さくしても**厚み（立体感）は絶対に残す**。「小さく」＝「薄く平たく」ではない。
- 🚫 文字を**浮かせない**（飛び出す・影で持ち上げない）。
- 🚫 表面に**ザラザラ質感を足さない**（色はクリーンなまま）。
- 🚫 **凹凸（レゴのポッチ・スタッド・突起）を付けない**。
  └ 「レゴみたい」は"可愛い雰囲気"の話。**ポッチを足す意味ではない**（前回ここを誤解した）。
- 🚫 **名前（タイトル）を一番大きくしない**。

## 4. サイズ（前回いちばん往復した所）

- ブロックは**今より小さく**。「100%で大きすぎ」と言われた → **全体を約0.7倍**に。
- 縮小は**一括で**やる。1か所ずつ大きくすると全体が膨らんで失敗する。
- ブロックの中の大きさの順番:
  - **一番大きい ＝ 操作する所**（スロット／フィールドの値）。子どもが触る所が主役。
  - **小さい ＝ 名前**。脇役でいい。
- 初期配置：ブロックは**画面の一番下**に出す。

## 5. ブロックの形（ブロック別）

### 条件分岐（緑）＝ 横長
3つのスロットが**横にゆったり1行ずつ**並ぶ。縦に詰め込まない。
```
┌──────────────────────────────┐
│           条件分岐            │ ← 名前は小さめ
│ ┌──────────────────────────┐ │
│ │ 🔷 もしも     [ 条件を置く ]│ │ ← 操作部=大きい・読める
│ │ ⚡ そうなら   [ 動作を置く ]│ │
│ │ ⚡ ちがうなら [ 動作を置く ]│ │
│ └──────────────────────────┘ │
└──────────────────────────────┘
        ↑ 横長（縦長にしない）
```

### アイテム付与（青）＝ 中身ピッタリ
中身（アイテム／個数）に対して**無駄な余白を作らない**。前は高さがデカすぎた。
```
┌─────────────────────────┐
│        アイテム付与       │ ← 名前は小さめ
│ アイテム [ 💎 ダイヤモンド ]│ ← 値=大きい・読める
│ 個数    [ 1 ]            │
└─────────────────────────┘
   ↑ 中身ピッタリ（余白を切る）
```
- アイテムは**アイコン＋日本語**（例: `💎 ダイヤモンド`）。生ID（`minecraft:diamond`）は出さない。変換表は `data/itemNames.ts`。

## 6. 触って楽しい（操作の気持ちよさ）

- **条件分岐のスロットを押す → そのスロット色の「配線ライン」が出る → 繋ぎたいブロックを押すと繋がる。**
- 繋がったら「**1つの塊**」に見えること（突起ではなく、**光と整列**で。子ブロックが親を隠さない）。
- 繋がる瞬間は **カチッ＋ひかり**（冒険のワクワク）。

## 7. 文字

- 可愛いフォント、または埋め込み（彫り込み）。浮かせない。読みやすさは絶対。

## 8. 完成チェック（実装後、ヒマワリが自分で確認 → OKならスクショ）

往復を減らすため、見せる前に自分でチェック:
- [ ] ブロックが「マイクラっぽい可愛いブロック」になってる（石板・遺跡っぽくない）
- [ ] 100%でブロックが大きすぎない（前の68%くらい）
- [ ] 操作部（スロット／値）が一番大きい。名前は脇役
- [ ] 名前が可愛く書いてある or 埋め込まれてる＆読める（浮いてない）
- [ ] 凹凸（ポッチ）なし／ザラザラ質感なし／色そのまま／角ばった形
- [ ] 条件分岐＝横長／アイテム付与＝無駄にデカくない
- [ ] アイテムが `💎 ダイヤモンド` 表示
全部 ✓ になってからスクショ。違う所は直してから見せる。

## 10. たいまつ（松明）の見た目（2026-06-05 ユーザー指示）

- **ドット絵にしない**。**本物のたいまつのようなリアルな見た目**にする。
- 配置: **ブロックが中心に来るように、両端から中心へ向かって（手前→奥のように）だんだん小さくなっていく**ように並べる（遠近・奥行きが出る置き方）。
- ※実装はヒマワリ（codebaseでたいまつの場所を確認のうえ）。

## 11. 操作部の質感＆タイトル（2026-06-05 ユーザー指示）

**狙い: 「押すもの(凸)」と「入力するもの(凹)」を見た目で区別する。**
- **タイトル(ブロック名)のフォント**: 少し大きく（今ちょい小さい）。ただし操作部より目立たせすぎない。
- **スロット = 完全に"押すボタン"の見た目＋感触**: 浮き出た立体ボタン（凸）、**押すと沈む**(:active / onMouseDown で translateY)、ホバーで軽く反応。「これは押すもの」と一目で分かる手触り。
- **入力フィールド = 差別化して"埋め込み(凹)"を強める**: 凹んで見える（inset shadow を強め）＝「ここに入力する」感。ボタン(凸)と入力(凹)で、触り方が見た目で分かるように。

## 12. 「正面が貼り付けパネルに見える」問題の直し方（2026-06-06 シオン分析）

**症状**: ブロックの正面が「ブロックに板を1枚貼り付けた」ように見えて、立体(キューブ)が"1つの塊"に見えない。
**原因（コード上の事実）**: 正面の面(front face)が、**4隅すべて角丸(borderRadius:R) ＋ 全辺にborder**を持つ「独立したカード」になっている → zIndex:2で上の面/横の面の手前に浮いて、板を貼ったように見える。

**直す方向（数値はヒマワリ感覚でOK）**:
- 正面とトップ面が**接する辺(上)・正面と横面が接する辺(右)**では、正面の**角丸と縁取りを消す**。継ぎ目で3枚が**融合して1つの塊**になるように。
- 角丸は**表に出る角（左下など、他の面と接しない角）だけ**に残す。継ぎ目側は角ばらせる。
- 立体感(厚み)は**上面・横面が担う**。正面は自前のborderで枠を主張しない＝「面が3枚」ではなく「キューブが1個」に見せる。
- ※P8の「繋がると1つの塊」と同じ思想（継ぎ目で融合）。

## 9. 約束ごと

- 細かい見た目（角丸・影・余白の数値）は**ヒマワリの感覚に任せる**。ルールとチェックだけ守る。
- 触らないロジック: `renderSingleArm`（条件スロット）は作り直してOK。ただし最終的に動くこと → `thenId/nextId` / `genBlock`(コード生成) / `getDepth` / `findSnap`。
- git: 作業ごとに必ずコミットして `git log` で実体確認（「コミットした」報告だけで終わらせない）。

## 1. 何を解決するか

旧モデルは「ノードを線でつなぐ」だけ。これだと **if/loop の“中身”と“その後”が区別できない**
（線が全部同じ顔 → メッセージ送信が「もし晴れなら」やるのか、if後に必ずやるのか不明）。

→ **if と loop はドーナツ（穴あき）にして、穴の中＝その制御の支配下** にする。
線の意味を覚えなくても「中に入ってる＝中身」が形で分かる。

---

## 2. 操作：吸い込みドッキング（ヒマワリ）

ノードをドラッグしてドーナツに近づけ、穴に**吸い込む**ことで「中身」にする。

**① 近づく（吸引）**
- ドーナツの引力圏（穴の中心から半径 R ≒ オーブ半径×1.8 目安）にドラッグ中ノードが入ったら、
  穴が**ぷっと開く（絞り/瞳孔が開く）**＝ドロップ可サイン。
- 穴へ向かう**光の筋がスパイラルで吸い込まれる**微エフェクト。ノードもわずかに穴へ引かれる。

**② 離す（スポッと収納）**
- 圏内でドロップ → ノードがキュッと縮んで穴へ着地（squash&stretch）。
- 着地でリングが**水滴状の波紋ショックウェーブ**。吸ったノードの色がリングに一瞬**色移り**。

**③ 取り出す（射出）**
- 中身を圏外へドラッグ → **スポッと抜けて**リングが縮む。

**if ドーナツだけ：穴を 真/偽 で二分割**
- ドラッグ中ノードが穴の**左寄り→真(緑に発光)／右寄り→偽(赤に発光)**。
- 光ってる側でドロップ → その枝に入る。**入れる前にどっちか分かる**＝誤爆しない。
- （左右は好み次第で上下でも可。データ上は then/else なので見た目は自由）

世界観メモ：背景が宇宙/星雲なので**ドーナツ＝小さな特異点（ワームホール）**。吸い込み＝自然。
子供っぽくせず「吸い込まれる気持ちよさ」で。

## 3. データ契約（★ここが見た目とコード生成の境界）

**吸い込み＝親子確定。** ノードを穴に入れたら、その `Fruit` に次をセットするだけでよい：

```ts
// GrapePanel の Fruit に追加（どちらも任意）
parentId?: string | null;          // 入っているドーナツの id。穴の外＝null/未設定
slot?: "then" | "else" | "body";   // if: then(真)/else(偽)、loop: body(中身)
```

- loop に吸い込んだ → `parentId = ドーナツid, slot = "body"`
- if の真側に吸い込んだ → `slot = "then"` ／ 偽側 → `slot = "else"`
- 引っ張り出した → `parentId`/`slot` を消す（トップレベルに戻る）

これだけで **シオン側が自動で正しい Java を生成**する（co_if の then/else、ct_rep の body）。
**位置(x,y)で判断したり、線を張ったりは不要**。親子は parentId/slot が唯一の真実。

### 構造導出は共有ライブラリを使う（二重実装しない）
ツリーの自動レイアウトも、この親子構造から組んでほしい。**同じ関数を import**：

```ts
import { buildGroveStructure } from "../../lib/groveTree";
const { roots, childrenOf } = buildGroveStructure(fruits);
// roots         … トップレベルの実（born順）= 幹
// childrenOf(id,"then"/"else"/"body") … その穴の中身（born順）= 枝
```

レイアウト（誰が誰の下/中か）とコード生成（thenId/elseId）が**同じ構造を見る**ので、見た目と動作がズレない。

## 4. シオン側 実装済み（このコミットに含む）

- `lib/groveTree.ts` … 親子構造導出の正本（`buildGroveStructure` / `GroveSlot`）。
- `lib/grapeToCBlock.ts` … parentId/slot を読んで
  - if: `slot=then→thenId(真)` / `slot=else→elseId(偽)` / 条件→innerId
  - loop: `slot=body→thenId(中身)` / 回数→フィールド n
  - 入れ子（ドーナツ in ドーナツ）も再帰的に成立。
  - **後方互換**：どの実も parentId を持たない間は従来動作のまま（吸い込みUI実装前でも壊れない）。
- 検証済み：旧フラット互換 / 吸い込み / else枝 / loop body の4ケースで期待通りの thenId/elseId/body を確認。

→ ヒマワリは **2章の操作** と **3章で parentId/slot をセットする** だけ。コード生成側は完成済み。
